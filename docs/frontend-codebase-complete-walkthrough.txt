Frontend Codebase Deep Dive

1. Overall Architecture and Design Patterns

This frontend is a Next.js 16 App Router application that serves:
- public marketing pages (portfolio)
- public content pages (blog, case studies, architecture notes)
- authenticated dashboard/CMS pages

The architecture intentionally follows a Backend-for-Frontend (BFF) model:

Browser -> Next.js page/client component -> same-origin Next.js /api route -> FastAPI backend -> repositories/services -> database

Key patterns used:
- Server Components by default, Client Components only when interactivity is required.
- Contract-first API design through TypeScript contracts in src/lib/api/contracts.
- Layered transport design:
  - contracts -> endpoint wrappers -> http client -> route handlers/services
- Edge middleware coordination in src/proxy.ts for:
  - dashboard auth gating
  - locale detection/normalization and path rewriting
- Feature and platform separation:
  - src/features for page-level domain compositions
  - src/components for reusable UI/layout pieces
  - src/lib for non-UI platform logic

5. State Management and Data Flow

Primary strategy:
- server-first data loading for public pages
- local component state for interactive dashboards

Data flow layers:
1) UI pages/components call service modules
2) service modules call endpoint wrappers
3) endpoint wrappers call the shared http client
4) http client resolves URL, applies timeout/auth, parses errors
5) backend returns typed JSON aligned with contracts

Client state examples:
- dashboard content workspace:
  - selected type/locale/status filters
  - deferred query value
  - item list/overview/load/error flags
- content editor:
  - form state
  - autosave timestamps
  - local draft persistence in localStorage

No global state manager is used, which is consistent with server-first rendering and route-local concerns.

8. API Integration and Network Design

Network strategy:
- public reads can use endpoint wrappers and service modules directly
- dashboard writes/reads go through same-origin API routes

Why this BFF design matters:
- browser remains independent from backend topology
- token handling stays server-side
- request shaping/logging/rate-limit hooks can be added centrally later

Shared http behavior in http-client.ts:
- AbortController timeout (default 10s)
- token-based Authorization header support
- ApiError abstraction with parsed backend message
- 204 response handling

9. Authentication and Authorization Flow

Login flow:
1) user posts credentials to /api/auth/login
2) route forwards to backend /auth/login
3) response access token stored as HttpOnly cookie (dashboard_access_token)
4) client receives user profile payload

Session usage:
- middleware checks cookie presence for dashboard route gating
- server session helper validates token against backend session endpoint
- unauthorized outcomes resolve to redirect/login or 401 in API layer

Logout:
- /api/auth/logout clears cookie with maxAge 0

Security posture:
- HttpOnly cookie (token inaccessible to client JS)
- SameSite=lax and secure in production
- backend remains source of truth for authorization

10. Form Handling and Validation Strategy

Current approach:
- controlled React inputs with typed form state
- backend-driven validation and detailed error parsing
- helper transforms for slug generation/csv parsing

Editor-specific behavior:
- auto-generate slug from title until user manually edits slug
- autosave form draft to localStorage (debounced)
- structured metadata rows are dynamic key/value entries

Validation note:
- frontend validation is intentionally light; backend enforces strict schema.

11. Styling System

Tailwind + tokenized CSS variables in globals.css.

Highlights:
- layered near-black surfaces
- strong text contrast hierarchy
- custom spacing/radii/duration/easing tokens
- accessible focus and skip-link styling
- prefers-reduced-motion support in CSS and JS
- RTL support for Arabic locale

No Styled Components or CSS-in-JS runtime is used.

12. Error Handling Strategy

Error control layers:
- app-level error boundary screen
- not-found page
- API error normalization into ApiError
- user-facing inline errors in dashboard workflows
- optimistic delete rollback logic in workspace
- graceful service fallbacks (returning empty arrays/null when backend unavailable)

13. Performance Optimizations and Best Practices

- server components reduce client bundle pressure
- fonts loaded via next/font with swap
- motion respects reduced-motion preference
- request-level cache for dashboard session resolver
- deferred search input for responsive filtering
- passive scroll listener + rAF throttling hook
- centralized motion tokens prevent expensive ad-hoc animation misuse

15. Deployment Flow and CI/CD Overview

Current repository supports local containerized/dev workflows with Docker and monorepo scripts. Typical production path:
- build with next build
- run with next start
- backend deployed independently
- web app points to backend via environment variables

Because authenticated calls are proxied through same-origin Next API routes, frontend deployment remains stable even if backend topology changes.

16. Frontend Security Considerations

19. Scalability and Maintainability

Why this scales:
- contracts and endpoint wrappers isolate backend changes
- service layer isolates data-source swaps
- content registry centralizes collection taxonomy
- i18n config centralizes locale behavior
- reusable UI primitives keep visual/system consistency

Adding a new content type usually requires:
1) extend content contract union
2) add registry entry
3) add route + page composition
4) optionally add dashboard filtering/support
